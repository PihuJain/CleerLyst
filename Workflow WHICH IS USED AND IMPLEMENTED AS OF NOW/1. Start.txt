📘 CLEERLYST — PRODUCT SPECIFICATION (V1)

## 1. What Cleerlyst Is (Core Idea)

**Cleerlyst is a privacy-first institutional verification platform.**

It replaces:

* PDFs
* Excel shortlists
* Google Sheets links
* Mass emails with attachments

With:

* identity-based access
* row-level visibility
* result-only notifications
* zero bulk data exposure

> **Principle:**
> *Institutions publish decisions. Individuals see only what applies to them.*

Cleerlyst is **NOT**:

* a placement portal
* a social app
* a document sharing tool

It is an **institutional data boundary**.

---

## 2. The Problem Cleerlyst Solves

Current institutional behavior:

* Broadcasts full datasets to thousands
* Leaks names, emails, reg numbers
* Enables scams & phishing
* Creates confusion, panic, and inbox overload

Cleerlyst fixes this by:

* Eliminating list sharing
* Eliminating attachments
* Eliminating search-based access
* Enforcing identity-first verification

---

## 3. Target Users

### A. Institutions / Admins

* Placement office
* Academic office
* Fest organizers
* Examination cells
* Finance / fee office
* Any authority that publishes lists

### B. Students / End Users

* Anyone who currently:

  * downloads PDFs
  * searches their name in Excel
  * forwards attachments
  * misses deadlines due to email overload

---

## 4. Product Philosophy (Non-Negotiable)

1. **No bulk visibility. Ever.**
2. **No searching datasets.**
3. **No attachments sent to students.**
4. **Identity determines access.**
5. **Minimize data exposure by default.**
6. **Admins should not need training.**

If a feature violates any of these, it does not ship.

---

## 5. Core Objects (Data Model – Conceptual)

### User

* id
* role: `student | admin`
* institute_domain
* verified_email
* optional identifiers (reg no, roll no)
* notification preferences

### Institute

* name
* domain(s)
* SSO config
* admin users

### Dataset (List / Sheet / PDF)

* uploaded_by (admin)
* institute_id
* type (placement, academic, fest, finance, other)
* source (internal / external like NPTEL)
* identifier_column (email / reg no)
* visibility_rules
* expiry_date
* status (draft / published / revoked)

### Record (Row)

* hashed_identifier
* encrypted_payload
* visibility_scope

### Notification

* user_id
* dataset_id
* type (new / update / action_required)
* read_status

## 6. Admin Workflow (Step-by-Step)

### Admin Goal

> Publish information **without leaking data** and **without chaos**.

### Admin Flow

1. **Login**

   * Institute email only
   * Domain-restricted

2. **Create Dataset**

   * Upload:

     * Excel / CSV / PDF
   * Platform parses file
   * Admin maps:

     * identifier column (email / reg no)
     * visible fields
     * hidden fields

3. **Configure Rules**

   * Who can see this dataset
   * What students can see
   * Enable:

     * no-download mode
     * watermarking
     * phased reveal (optional)
   * Set expiry date

4. **Publish**

   * Cleerlyst generates:

     * one universal link (optional)
     * push notifications to affected users
   * No files are sent

5. **Monitor**

   * View stats:

     * users matched
     * users viewed
     * errors (missing identifiers)

6. **Update / Revoke**

   * Update dataset → students notified
   * Revoke dataset → access instantly removed

---

## 7. Student Workflow (Step-by-Step)

### Student Goal

> Know what applies to me, instantly, without hunting or leaking data.

### Student Flow

1. **Install App / Open Web**

   * Android / iOS / Web
   * One-time login with institute email

2. **Identity Setup (Once)**

   * Verify email
   * Add optional identifiers (reg no)
   * Stored securely (hashed)

3. **Home Feed**

   * Timeline of updates:

     * “Dell Technologies – Shortlisted”
     * “NPTEL Enrollment – Action Required”
     * “Fest Payment – Confirmed”

4. **Open Item**

   * See:

     * status (yes / no)
     * instructions
     * deadlines
   * Cannot:

     * download files
     * view others’ data
     * see list size

5. **History**

   * Immutable log of:

     * what was published
     * when it was viewed

6. **Notifications**

   * Push notifications (result-only)
   * No sensitive data on lock screen

## 8. Mobile App Strategy (Explicit)

### Tech Strategy

* **Web-first (PWA)**
* Wrapped using:

  * Expo / Capacitor / React Native WebView

### Why

* One codebase
* Faster iteration
* Lower cost
* Sufficient performance

### App Capabilities

* Secure login
* Push notifications
* Feed view
* Pull-to-refresh
* Session-based access (no local storage)

---

## 9. Security Model (Very Important)

### What Cleerlyst Guarantees

* No bulk export of readable data
* Row-level encryption
* Identity-bound access
* Audit logs for admins
* Automatic expiry of datasets

### What Cleerlyst Does NOT Promise

* Absolute screenshot prevention
* Device-level security

### Security Assumptions

* Devices can be compromised
* Emails can be forwarded
* Users can behave badly

Design assumes **hostile environments**.

---

## 10. Anti-Scam & Trust Features

* All datasets tagged as:

  * Verified institutional
  * External reference
* Students can verify:

  * whether an email/message is legit
* No unofficial lists allowed

---

## 11. What Cleerlyst Explicitly Does NOT Build (V1)

❌ Chat
❌ Resume builders
❌ Ranking systems
❌ AI recommendations
❌ Social features
❌ Payments

Focus is **distribution + verification**, nothing else.
