1️⃣ DATABASE SCHEMA — CLEERLYST (V1)

> Goal: **No bulk access, no plaintext identity lookup, no accidental joins**

Assume **PostgreSQL**.

---

## 1.1 Core Tables

### `institutes`

```sql
id (uuid, pk)
name (text)
primary_domain (text)
allowed_domains (text[])
created_at
```

---

### `users`

```sql
id (uuid, pk)
institute_id (fk)
role (enum: student, admin)
email_hash (text, unique)
email_verified (boolean)
created_at
last_login_at
```

❗ RULES:

* NEVER store plaintext email for lookup
* email_hash = SHA-256(lowercase(email + institute_salt))

---

### `user_identifiers`

```sql
id (uuid, pk)
user_id (fk)
type (enum: reg_no, roll_no, employee_id)
identifier_hash (text, unique)
created_at
```

Allows multiple identifiers per user
(e.g. email + reg number).

---

## 1.2 Dataset & Records (CRITICAL)

### `datasets`

```sql
id (uuid, pk)
institute_id (fk)
created_by (admin user_id)
type (enum: placement, academic, fest, finance, other)
title (text)
description (text)

identifier_type (enum: email, reg_no)
visibility_config (jsonb)
expires_at (timestamp)

status (enum: draft, published, revoked)
created_at
published_at
```

`visibility_config` example:

```json
{
  "allowed_fields": ["status", "venue", "time"],
  "phased_reveal": true,
  "no_download": true
}
```

---

### `dataset_records`

```sql
id (uuid, pk)
dataset_id (fk)

identifier_hash (text)
encrypted_payload (bytea)

created_at
```

❗ ABSOLUTE RULES:

* One row = one record
* No foreign key to users
* No joins allowed from users → records
* Lookup ONLY via identifier_hash

---

## 1.3 Notifications & Audit

### `notifications`

```sql
id (uuid, pk)
user_id (fk)
dataset_id (fk)
type (enum: new, update, action_required)
read_at (timestamp)
created_at
```

---

### `audit_logs`

```sql
id (uuid, pk)
actor_user_id (fk)
action (text)
dataset_id (fk, nullable)
metadata (jsonb)
created_at
```

Log actions, **never payloads**.

---

## 1.4 Forbidden Schema Patterns ❌

* ❌ `SELECT * FROM dataset_records`
* ❌ foreign key from record → user
* ❌ storing plaintext email/reg no
* ❌ materialized views over records

If AI generates these → reject.

# 2️⃣ THREAT MODEL — ATTACKER POV

This is what makes Cleerlyst **actually secure**.

---

## 2.1 Threat Actors

### A. Curious Student

* Wants to see who else is shortlisted
* Tries guessing names / reg numbers

### B. Malicious Student

* Writes a scraper
* Iterates identifiers
* Tries brute-force

### C. Insider Admin

* Tries exporting data
* Misconfigures visibility

### D. External Attacker

* Phishing
* Fake placement emails
* Credential stuffing

---

## 2.2 Attack Vectors & Defenses

---

### Attack: Dataset Enumeration

> “Let me query many identifiers and reconstruct the list”

**Defense**

* No search endpoints
* Identity-bound access only
* Backend ignores arbitrary identifiers
* Rate limiting

✅ Attack fails.

---

### Attack: Scraping via Automation

> “I’ll open thousands of sessions and collect responses”

**Defense**

* Login required
* One identity → one record
* Same response shape for yes/no
* No list size leakage

✅ Scraper gets nothing useful.

---

### Attack: Screenshot & Forwarding

> “I’ll screenshot and leak”

**Defense**

* Watermark (email + timestamp)
* No downloadable files
* Psychological + accountability deterrent

⚠️ Cannot fully prevent
✅ Can strongly discourage

---

### Attack: Admin Mistake

> “Admin uploads wrong file / exposes fields”

**Defense**

* Visibility mapping step
* Default-minimal exposure
* Warning on sensitive fields
* Audit logs

✅ Damage minimized.

---

### Attack: DB Breach

> “Attacker gets DB dump”

**Defense**

* Identifiers hashed
* Payloads encrypted
* No join path
* No bulk readable data

✅ Breach is useless.

---

### Attack: Phishing

> “Fake Dell mail asking for fee”

**Defense**

* Students trust Cleerlyst as source of truth
* All official actions appear in app
* Anything outside = suspicious

✅ Scam effectiveness drops.

---

## 2.3 Non-Goals (Be Honest)

Cleerlyst does NOT:

* protect compromised devices
* stop intentional leaks
* replace endpoint security

It **reduces institutional blast radius**.

# 3️⃣ API CONTRACTS — SECURITY-FIRST

> These are **the only APIs that should exist**.

---

## 3.1 Auth

### `POST /auth/login`

```json
{
  "email": "student@vitbhopal.ac.in"
}
```

Response:

```json
{
  "session_token": "jwt"
}
```

---

## 3.2 Admin APIs

### Upload Dataset

`POST /datasets`

```json
{
  "title": "Dell Technologies Online Test",
  "type": "placement",
  "identifier_type": "email",
  "expires_at": "2026-02-01"
}
```

---

### Upload Records

`POST /datasets/{id}/records`

```json
{
  "records": [
    {
      "identifier": "student@vitbhopal.ac.in",
      "payload": {
        "status": "shortlisted",
        "venue": "LC-101",
        "time": "6:00 PM"
      }
    }
  ]
}
```

⚠️ Backend:

* hashes identifier
* encrypts payload
* discards plaintext

---

### Publish Dataset

`POST /datasets/{id}/publish`

No payload.

---

## 3.3 Student APIs

### Feed

`GET /me/feed`

```json
[
  {
    "dataset_id": "uuid",
    "title": "Dell Technologies",
    "type": "placement",
    "status": "available",
    "created_at": "timestamp"
  }
]
```

---

### Get My Record

`GET /datasets/{id}/me`

```json
{
  "matched": true,
  "data": {
    "status": "shortlisted",
    "venue": "LC-101",
    "time": "6:00 PM"
  }
}
```

If not matched:

```json
{
  "matched": false
}
```

❗ Same HTTP status, same shape → prevents inference.

---

## 3.4 Forbidden APIs ❌

These must NEVER exist:

* ❌ `/datasets/{id}/records`
* ❌ `/search`
* ❌ `/export`
* ❌ `/download`
* ❌ `/users/{id}/records`

If an AI suggests one → reject immediately.
