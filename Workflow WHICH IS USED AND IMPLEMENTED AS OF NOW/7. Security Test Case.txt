🔐 CLEERLYST — SECURITY TEST CASES

**(Attacker-Driven, Non-Negotiable)**

Each test has:

* **Threat**
* **Attack attempt**
* **Expected result**
* **Failure meaning**

If *any* test fails → the system is **not Cleerlyst**.

---

## 🟥 CATEGORY 1: IDENTITY & AUTHENTICATION

### Test 1.1 — Domain Restriction

**Threat:** External user joins system
**Attack:**

* Try login with `gmail.com`
  **Expected:**
* Login denied
* No user record created
  **Failure Means:**
* Anyone can probe datasets → fatal

---

### Test 1.2 — Email Hashing

**Threat:** Insider DB leak
**Attack:**

* Inspect `users` table
  **Expected:**
* No plaintext email
* Only hashes present
  **Failure Means:**
* Database leak becomes actionable → fatal

---

### Test 1.3 — Role Escalation

**Threat:** Student accessing admin routes
**Attack:**

* Student visits `/admin/datasets`
  **Expected:**
* 403 / redirect
  **Failure Means:**
* Students can manipulate datasets → fatal

---

## 🟥 CATEGORY 2: DATASET ACCESS (MOST IMPORTANT)

### Test 2.1 — No Dataset Listing

**Threat:** Dataset enumeration
**Attack:**

* Call `/api/datasets/{id}/records`
  **Expected:**
* Endpoint does not exist
  **Failure Means:**
* Entire list can be scraped → fatal

---

### Test 2.2 — Identity-Bound Access

**Threat:** Student guessing another identifier
**Attack:**

* Student A tries to fetch dataset using Student B’s reg no
  **Expected:**
* Backend ignores input
* Only logged-in identity used
  **Failure Means:**
* Scraping becomes possible → fatal

---

### Test 2.3 — Non-Match Response Uniformity

**Threat:** Inference attack
**Attack:**

* Compare response when matched vs not matched
  **Expected:**
* Same HTTP status
* Same response shape
  **Failure Means:**
* Attackers can infer list size → severe

---

### Test 2.4 — No Join Path

**Threat:** Backend dev mistake
**Attack:**

* Attempt SQL join between `users` and `dataset_records`
  **Expected:**
* Impossible by schema design
  **Failure Means:**
* Bulk access becomes trivial → fatal

---

## 🟥 CATEGORY 3: SCRAPING & ENUMERATION

### Test 3.1 — No Search APIs

**Threat:** Automated scraping
**Attack:**

* Look for endpoints like `/search`, `/lookup`, `/find`
  **Expected:**
* None exist
  **Failure Means:**
* List reconstruction possible → fatal

---

### Test 3.2 — Rate Limiting

**Threat:** Brute force identity probing
**Attack:**

* Spam `/datasets/{id}/me`
  **Expected:**
* Rate limited
* Temporary block
  **Failure Means:**
* Scraper can iterate accounts → high risk

---

### Test 3.3 — Same-User Repetition

**Threat:** Scraper using one account
**Attack:**

* Same user calls API repeatedly
  **Expected:**
* Same single record returned
  **Failure Means:**
* Multiple rows being exposed → fatal

---

## 🟥 CATEGORY 4: ADMIN MISUSE & MISTAKES

### Test 4.1 — Admin Cannot Preview Rows

**Threat:** Curious admin
**Attack:**

* Admin tries to view dataset contents

* No row preview UI

  **Failure Means:**
* Insider leak risk → fatal

**Threat:** Accidental exposure

  **Failure Means:**
* Easy human error → severe

### Test 4.3 — Dataset Revocation

  **Expected:**

  **Failure Means:**
* Damage can’t be contained → severe

## 🟩 PASS CRITERIA (THIS IS STRICT)

Cleerlyst is considered **secure enough to ship** only if:

* ✅ No test marked **fatal** fails
* ⚠️ “Severe” issues are acknowledged + mitigated
* ❌ Any fatal failure → stop shipping immediately

---

## 🔑 How to Use This Practically

### With AI Coders

Paste the relevant test before asking them to build a feature:

> “Implement X while ensuring Test 2.1, 2.2, 2.3 still pass.”

### With Yourself

Before merging:

> “Which test does this change threaten?”

If you can’t answer → don’t merge.
